                     Debrief 
__________________________________________________
  
I focused on creating an immersive experience by making the environment feel alive through smooth camera movement, lighting, and subtle animated effects, which helped improve the overall flow and player engagement.
* Improvement: The code needs cleaner organisation. I believe my goal of making sure the values were easy to find resulted in overcomplicating the global variables.

I believe the structured game state system (main menu, game, instructions, pause) helped improve the  organisation in the code and makes it easier to control what is displayed at any time.
* Improvement: the modular design wasn’t fully executed well, as I found it easier to manually set values to have better control over  placement (e.g., the buttons), which may make it difficult to adapt to different screen sizes or resolutions

The use of pixel art combined with a cold, minimal visual style helps reinforce the game’s atmosphere and theme, making the experience feel more surreal and intentionally lonely.
* Improvement: Adding more visual variation or animation would push the visuals further. For example, different enemies could symbolise various negative emotions or worries, and animating the player could show responses to damage (such as visual changes to the ship when hit).

The enemy spawning system enabled scalable difficulty and controlled pacing.
* Improvement: The difficulty progression (increase in challenge as the game advances) is currently static and could be improved with dynamic scaling based on player performance, which could even add depth to the theme of the game.

   Game State System
-----------------------
  let gameState = "mainMenu";              // Created a variable that controls the entire game’s “mode”:    
  if (gameState !== "mainMenu") return;   // Then in your functions you check it like this:

Each part of the game runs only when its state is active (e.g., mainMenuState() runs in the menu), acting as a traffic controller. This prevents overlapping code and mirrors standard industry systems. 



   Tethered Player (Rope / Constraint Physics System)
--------------------------------------------------------
Combination of lerp smoothing (soft movement), distance constraint (hard limit), procedural rope rendering (math-based animation)

this.x = lerp(this.x, targetX, this.lerpAmount); // Smooth following - creates inertia-like motion instead of instant snapping.
this.y = lerp(this.y, targetY, this.lerpAmount);

if (d > this.maxDistance) { let angle = atan2(this.y - anchorY, this.x - anchorX); // Rope constraint
this.x = anchorX + cos(angle) * this.maxDistance; this.y = anchorY + sin(angle) * this.maxDistance; } // It calculates the direction from the spaceship → player, and if too far, it pulls the player back onto the circle boundary.

------------------------------

Instead of designing separate screens, I developed a single vertical world. I chose this approach because it required less coding than implementing dynamic button movement, which prevents buttons from abruptly appearing or disappearing during transitions. This simplifies side-scrolling engines and camera systems in Unity.
